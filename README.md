# Отчет по практическим работам №13–17
## Разработка офлайн-приложения для заметок (PWA)

В ходе выполнения работ **№13–17** создано прогрессивное веб-приложение (PWA), которое устанавливается на устройство, работает офлайн и отправляет уведомления с возможностью отложенного повтора.

---

### Практическая работа №13 — Service Worker

**Цель:** Реализовать базовый офлайн-функционал с использованием Service Worker.

**Результат:**
- Зарегистрирован Service Worker.
- Статические файлы (HTML, CSS, JS, иконки) закэшированы.
- Применена стратегия `Cache First`: при офлайне интерфейс загружается из кэша мгновенно.

---

### Практическая работа №14 — Web App Manifest

**Цель:** Добавить поддержку Web App Manifest и возможность установки на устройство.

**Результат:**
- Создан `manifest.json`.
- Добавлен набор иконок (16x16, ..., 512x512)
- Добавлены мета-теги для поддержки iOS (Apple Touch Icon) и Android.
- Приложение доступно для установки.

---

### Практическая работа №15 — HTTPS и App Shell

**Цель:** Обеспечить безопасный канал передачи данных и ускорить загрузку приложения.

**Результат:**
- Настроен локальный HTTPS-сервер с использованием `mkcert`.
- Внедрена архитектура **App Shell**: оболочка интерфейса кэшируется сразу, данные заметок подгружаются динамически.

---

### Практическая работа №16 — WebSocket и Push-уведомления

**Цель:** Реализовать синхронизацию данных в реальном времени и отправку системных уведомлений.

**Результат:**
- Реализован WebSocket-сервер на `Socket.IO` для мгновенной рассылки событий.
- Настроена подписка на push-уведомления через `web-push`.
- Добавлен переключатель для управления подпиской.

---

### Практическая работа №17 — Отложенные напоминания

**Цель:** Добавить функционал планирования заметок на конкретную дату и время с возможностью переноса.

**Результат:**
- Реализована форма выбора даты и времени.
- На сервере через `setTimeout` реализован отложенный запуск уведомлений.
- В push-сообщение добавлена кнопка **«Отложить»**, при клике на которую таймер напоминания переносится на 5 минут.

---

### Используемые технологии
- **Клиент:** HTML/CSS/JS, Service Worker.
- **Сервер:** Node.js, Express, Socket.IO, Web-Push.
- **Среда:** HTTPS (mkcert), PWA (Manifest).

# Отчет о практических работах №7–11

## Введение

В рамках практических работ **7–11** было последовательно разработано клиент-серверное веб-приложение с использованием **Node.js** для серверной части и **React.js** для клиентской части.

На каждом следующем этапе приложение дорабатывалось и расширялось: от базового REST API с регистрацией и товарами до полноценной системы с авторизацией, refresh-токенами, пользовательским интерфейсом и разграничением прав доступа по ролям.

---

## Практическая работа №7

### Цель
Создание базового серверного API на Node.js для работы с пользователями и товарами.

### Реализованные маршруты

| Маршрут | Метод | Описание |
|---------|-------|----------|
| `/api/auth/register` | `POST` | Регистрация нового пользователя |
| `/api/auth/login` | `POST` | Аутентификация пользователя |
| `/api/products` | `POST` | Добавление нового товара |
| `/api/products` | `GET` | Получение списка всех товаров |
| `/api/products/:id` | `GET` | Получение информации о конкретном товаре |
| `/api/products/:id` | `PUT` | Обновление данных товара |
| `/api/products/:id` | `DELETE` | Удаление товара |

### Модели данных

#### Пользователь
```javascript
{
  id: "string",           // Уникальный идентификатор 
  email: "string",        // Используется в качестве логина
  first_name: "string",   // Имя пользователя
  last_name: "string",    // Фамилия пользователя
  password: "string"      // Хеш пароля (bcrypt)
}
```

#### Товар
```javascript
{
  id: "number",           // Уникальный идентификатор
  title: "string",        // Наименование товара
  category: "string",     // Категория
  description: "string",  // Описание
  price: "number",        // Цена
  stock: "number",        //Кол-во товаров на складе
  image: "string"         //Путь к изображению
}
```

### Результат
Создано базовое серверное API, реализующее регистрацию и аутентификацию пользователей, а также полный набор CRUD-операций для управления товарами с поддержкой изображений.

---

## Практическая работа №8

### Цель
Добавление механизма авторизации на основе JWT-токенов и защищенных маршрутов.

### Основные доработки

1. **Генерация accessToken** при успешном входе пользователя
2. **Добавление защищенного маршрута** для получения информации о текущем пользователе

| Маршрут | Метод | Описание |
|---------|-------|----------|
| `/api/auth/me` | `GET` | Получение данных текущего пользователя |

### Защищенные маршруты

Следующие маршруты стали доступны только при наличии валидного accessToken:

- `GET /api/products/:id` — просмотр конкретного товара
- `PUT /api/products/:id` — обновление товара
- `DELETE /api/products/:id` — удаление товара

### Результат
Реализована система авторизации на основе JWT-токенов. Доступ к определенным маршрутам ограничен только аутентифицированными пользователями.

## Практическая работа №9

### Цель
Реализация механизма refresh-токенов для безопасного обновления сессии.

### Выполненные доработки

1. **Генерация refreshToken** вместе с accessToken при входе
2. **Добавление маршрута для обновления токенов**

| Маршрут | Метод | Описание |
|---------|-------|----------|
| `/api/auth/refresh` | `POST` | Получение новой пары токенов |

### Формат ответа при входе в систему

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

### Результат
Внедрена полноценная система аутентификации с использованием access и refresh токенов, повышающая безопасность приложения.

## Практическая работа №11

### Цель
Внедрение системы ролей и разграничение прав доступа для различных категорий пользователей.

### Реализованные роли

| Роль | Описание прав |
|------|---------------|
| **Гость** | Неаутентифицированный пользователь, имеет доступ только к страницам регистрации и входа |
| **Пользователь** | Может просматривать список товаров и их детальную информацию |
| **Продавец** | Имеет права пользователя + может создавать и редактировать товары |
| **Администратор** | Обладает всеми правами продавца + может управлять пользователями (просмотр, редактирование, блокировка) |

### Доработка модели пользователя

В Практике №11 модель пользователя была расширена двумя ключевыми полями:

- **`role`** ('user' | 'seller' | 'admin') — определяет права доступа
- **`isBlocked`** (boolean) — позволяет блокировать пользователей

### Маршруты и права доступа

| Маршрут | Метод | Доступ | Описание |
|---------|-------|--------|----------|
| `/api/auth/register` | `POST` | Гость | Регистрация пользователя |
| `/api/auth/login` | `POST` | Гость | Вход в систему |
| `/api/auth/refresh` | `POST` | Гость | Обновление пары токенов |
| `/api/auth/me` | `GET` | Пользователь | Получение текущего пользователя |
| `/api/users` | `GET` | Администратор | Получение списка пользователей |
| `/api/users/:id` | `GET` | Администратор | Получение пользователя по id |
| `/api/users/:id` | `PUT` | Администратор | Обновление информации пользователя |
| `/api/users/:id` | `DELETE` | Администратор | Блокировка пользователя |
| `/api/products` | `POST` | Продавец | Создание товара |
| `/api/products` | `GET` | Пользователь | Получение списка товаров |
| `/api/products/:id` | `GET` | Пользователь | Получение товара по id |
| `/api/products/:id` | `PUT` | Продавец | Обновление параметров товара |
| `/api/products/:id` | `DELETE` | Администратор | Удаление товара |

### Новые возможности клиентской части

#### Работа с товарами (доступно в зависимости от роли)
- Просмотр списка товаров (все авторизованные пользователи)
- Создание нового товара (продавец, администратор)
- Просмотр детальной информации (все авторизованные пользователи)
- Редактирование товара (продавец, администратор)
- Удаление товара (только администратор)

#### Работа с пользователями (только для администратора)
- Просмотр списка всех пользователей
- Просмотр детальной информации о пользователе
- Редактирование данных пользователя (имя, email, роль)
- Блокировка/разблокировка пользователей

### Результат
Приложение дополнено гибкой системой ролей и прав доступа, что позволяет реализовать различные сценарии работы для всех категорий пользователей: от гостей до администраторов.
